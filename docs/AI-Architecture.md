# Capture Inbox AI Architecture

## Purpose

This document describes the architecture behind the AI features of
Capture Inbox.

The guiding philosophy is:

> Treat AI as a pipeline, not as a feature.

The application should progressively transform a saved URL into
actionable knowledge.

------------------------------------------------------------------------

# High-Level Pipeline

``` text
Capture
    ↓
Extract
    ↓
Normalize
    ↓
Analyze
    ↓
Recommend
    ↓
Learn From Feedback
```

Each stage has a single responsibility.

------------------------------------------------------------------------

# Stage 1 --- Capture

Represents the user's action of saving something.

``` ts
interface Capture {
  id: string;
  url: string;
  source: "youtube" | "instagram" | "article";
  status: "INBOX" | "REVIEWED" | "ARCHIVED";
  createdAt: Date;
}
```

Responsibilities

-   Store the original URL.
-   Track lifecycle status.
-   Never contain AI-generated information.

------------------------------------------------------------------------

# Stage 2 --- Content Extraction

Each source implements an extractor that converts raw content into a
common format.

``` text
YouTube Extractor
Instagram Extractor
Article Extractor
PDF Extractor (future)
Reddit Extractor (future)
```

All produce the same output.

``` ts
interface ContentDocument {
  title?: string;
  description?: string;
  articleText?: string;
  transcript?: string;
  author?: string;
  publishedAt?: Date;
  duration?: number;
  thumbnail?: string;
  source: "youtube" | "instagram" | "article";
}
```

This stage performs **no AI reasoning**.

Its only job is normalization.

------------------------------------------------------------------------

# Stage 3 --- AI Analysis

The normalized document is sent to an LLM.

The LLM infers structured information.

``` ts
interface AIAnalysis {
  topics: string[];

  difficulty:
    | "beginner"
    | "intermediate"
    | "advanced";

  targetAudience: string[];

  contentType:
    | "tutorial"
    | "deep-dive"
    | "reference"
    | "news"
    | "opinion";

  containsCode: boolean;
  containsHandsOn: boolean;

  prerequisites: string[];

  estimatedMinutes: number;

  learningOutcomes: string[];

  summary: string;

  keyTakeaways: string[];

  reasoning: string;

  confidence: number;
}
```

The application now understands the content.

------------------------------------------------------------------------

# Stage 4 --- Recommendation

Recommendations are generated by combining:

-   ContentDocument
-   AIAnalysis
-   UserProfile

``` ts
interface Recommendation {
  score: number;

  action:
    | "read_now"
    | "later"
    | "skip";

  expectedValue: number;

  expectedEffort: number;

  confidence: number;

  reasoning: string[];
}
```

This layer answers:

> Is this worth my time?

------------------------------------------------------------------------

# Stage 5 --- User Profile

Recommendations become personal instead of generic.

``` ts
interface UserProfile {
  interests: string[];

  goals: string[];

  preferredDifficulty: string[];

  preferredContentTypes: string[];

  availableMinutesPerDay: number;
}
```

The same article can produce different recommendations for different
users.

------------------------------------------------------------------------

# Stage 6 --- Learning From Feedback

The application improves by observing behaviour.

Examples

-   opened
-   ignored
-   completed
-   archived
-   liked
-   time spent

``` ts
interface UserFeedback {
  captureId: string;
  opened: boolean;
  completed: boolean;
  ignored: boolean;
  liked: boolean;
  timeSpentMinutes: number;
}
```

# Processing Lifecycle

Track pipeline progress independently from capture status.

``` ts
interface CaptureProcessing {
  extractionStatus:
    | "pending"
    | "processing"
    | "completed"
    | "failed";

  analysisStatus:
    | "pending"
    | "processing"
    | "completed"
    | "failed";

  recommendationStatus:
    | "pending"
    | "processing"
    | "completed"
    | "failed";
}
```

------------------------------------------------------------------------

# Design Principles

-   AI should operate on normalized content.
-   Content extraction and AI analysis are separate concerns.
-   Structured data is more valuable than free-form summaries.
-   Recommendations should always explain *why*.
-   Every stage should be independently replaceable.
-   AI models can change without changing extraction.
-   New content sources should require only a new extractor.
-   The recommendation engine should remain source-agnostic.

------------------------------------------------------------------------

# Long-Term Vision

Capture Inbox should evolve through these stages:

``` text
URL
    ↓
Content
    ↓
Understanding
    ↓
Recommendation
    ↓
Learning
    ↓
Knowledge
```

The final objective is not to summarize everything.

It is to help users answer one question:

> **"Given everything I could consume today, what is actually worth my
> attention?"**
